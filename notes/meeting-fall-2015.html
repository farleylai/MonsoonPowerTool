ODROID XU3 Issues

There two critical issues with ODROID XU3 that I already posted on the forum as follows but there are no positive feedback or fixes so far. The first issue is that the available number of cores does not recover if once set offline. The workaround is not to set cores offline but in a batch test, the nproc is observed to decrease for some reason. The second issue is that running the 4 core version causes reboot at nearly highest frequencies in seconds. Depending the benchmark, no workaround so far but setting the frequencies conservatively.

  • nproc is never restored to the max available number of cores

  • Kernel reboots when running 8-core benchmarks

Performance and Energy Measurement against Single and Multi Core Versions

  • Depending on the benchmarks, some multi-core version may outperform the single-core version slightly at very high workload

    • Single-core AutoCor always performs the best even with 300X workload

    • Dual-core FFT2 starts to outperform the single-core version with 3X workload but the quad-core version always loses

  • For the SAME multi-core version, the more cores, the higher performance and lower energy consumption

  • The optimized single-core version is strong unless the thread synchronized overhead is reduced for multi-core versions

  • Previous work conducting experiments on the SAME compiled version makes little sense for stream programs optimized for a single-core

  • Allowing to schedule on both A7 and A15 does not improve musch

  • Cache misses seem to increase with the number of cores

Data-Oriented Characterization

  • Data access patterns

    • sequential/random read/write through index array

    • less randomness, less cache miss, higher performance, more savings

    • no significant energy consumption difference between read/writ

  • Data organization

    • primitive array vs boxed arraylist, object array vs arrays of primitives

    • boxed and access methods cause more energy consumption up to 2X

    • VQ: value access > TQ: type access > RQ: reference access

    • Object-centric > Attribute-centric for cache misses due to new objects

  • Data type precision

    • double > long > float = int > short

  • Data I/O strategies (Disk)

    • unbuffered > buffered

    • write > read
